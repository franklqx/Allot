# Project Time Planner Product Spec

## 0. Ship locale

The shipped app uses **English** for all user-facing copy (navigation, labels, buttons, notifications, empty states, formats). UI prototypes and marketing surfaces should match unless explicitly marked exploratory.

## 1. 产品名称

暂定名：Project Time Planner

内部定义：项目导向的时间分配与记录 App。

## 2. 产品定位

### 2.1 一句话定位

帮助独立开发者、自由职业者、多项目创业者，先规划今天的时间分配，再真实记录每段投入，最后回看自己把时间给了什么。

### 2.2 产品本质

它不是传统待办清单，也不是纯番茄钟。

它是一个：

“项目时间分配 + 任务执行记录 + 时间复盘”工具。

### 2.3 核心价值

用户能明确知道：

- 今天有哪些任务要做
- 每个任务属于哪个项目
- 我在每个任务上到底投入了多久
- 今天 / 本周 / 本月 / 本年，时间被哪些项目吃掉了
- 哪些时间没有被记录，需要补全

## 3. 目标用户

### 3.1 核心用户

- 独立开发者
- 自由职业者
- 多项目创业者
- 个人创作者

### 3.2 用户特征

- 同时推进多个项目
- 容易沉浸在单个任务里，忽略整体分配
- 想要简洁，不想学复杂系统
- 在意“真实投入时间”，而不是只看任务勾选完成

### 3.3 用户痛点

- 一天做了很多事，但说不清时间去哪了
- 只忙一个项目，其他重要事项被落下
- 想规划时间，但不想每次都写很复杂
- 事后很难复盘自己是否把时间给了真正重要的事

## 4. 产品目标

### 4.1 用户目标

- 规划今天的任务与时间
- 执行时快速开始 / 暂停 / 结束
- 支持遗漏后手动补录
- 看到各项目真实投入占比
- 通过历史与趋势调整时间策略

### 4.2 产品目标

MVP 阶段只做到：

- 简单
- 好看
- 能每天真的打开用
- 2-4 周可做出可测试版本

## 5. 最终产品结构

### 5.1 底部导航

只有两个主 tab：

- Home
- Insights

### 5.2 顶部导航栏

仅存在于 Home：

- 中间：当前日期
- 右边：Calendar 按钮

### 5.3 悬浮主按钮

右下角一个圆形液态玻璃按钮：

- 当前在今天：显示 `+`
- 当前不在今天：显示 `Today`

## 6. 信息架构

### 6.1 Home

职责：

- 查看当天任务
- 开始 / 暂停 / 结束记录
- 浏览附近日期
- 跳转到任意日期
- 查看某一天的执行情况
- 提醒补录未记录时段

### 6.2 Insights

职责：

- 看时间分配
- 看项目占比
- 看分类占比
- 看日 / 周 / 月 / 年趋势

### 6.3 Calendar

Calendar 不是独立页面，只是 Home 的日期选择弹层。

职责：

- 只做日期跳转
- 只服务于 Home
- 不承担分析功能

## 7. 核心设计原则

### 7.1 Home 是执行页

用户打开 App，大多数时候都应该落在 Home，因为 Home 才是高频操作页。

### 7.2 Insights 是分析页

Insights 不是让用户每天盯着图表，而是让用户定期检查：

- 我时间是不是分配对了
- 我是不是把精力都给错地方了

### 7.3 Calendar 只是导航器

Calendar 只负责日期跳转，不承担分析功能。

### 7.4 未记录时长必须保留

“未记录时长”必须保留，因为它能提醒用户补齐当天。

## 8. Home 页面 PRD

### 8.1 页面目标

Home 是执行、记录和查看具体某天的核心页面。

### 8.2 页面结构

#### A. 顶部导航栏

- 中间显示当前日期
- 右侧显示 Calendar 按钮

#### B. 日期滑动条

用于浏览附近日期。

#### C. 当天概览

显示当天执行情况，包括已记录时长与未记录时长。

#### D. AM 区块

标题：AM

内容：上午任务列表。

每个任务卡包含：

- 任务名
- 项目名，可选
- 已记录时长
- 目标时长或倒计时长度，可选
- 状态标签：未开始 / 进行中 / 已暂停 / 已完成

右侧操作：

- 开始
- 若已开始，则显示暂停 / 结束

逻辑：

- 用户创建任务时，如果没有选时间段，当前时间在上午则自动放到 AM
- 用户创建任务时，如果没有选时间段，当前时间在下午则自动放到 PM

#### E. PM 区块

标题：PM

结构同 AM。

#### F. 当前进行中任务卡

当存在计时中的任务时显示。

建议固定在较显眼位置，可以在 AM / PM 上方，也可以悬浮在列表上方。

内容：

- 任务名称
- 所属项目
- 当前模式：计时 / 倒计时
- 已进行时长 / 剩余时长
- 暂停按钮
- 结束按钮

这个模块很重要，因为用户打开 App 后最常问的是：

“我现在正在做什么？”

#### G. 补录提醒条

当系统发现当天有较长未记录时段时显示。

点击补录提醒或 Add Manual Record 进入补录界面。

### 8.3 Home 核心交互

#### 交互 1：开始任务

- 点击任务上的开始按钮
- 任务进入进行中状态
- 当前进行中任务卡出现

#### 交互 2：暂停任务

- 点击暂停
- 当前 session 暂停
- 任务状态变为已暂停

#### 交互 3：结束任务

- 点击结束
- 保存 session
- 任务可进入已完成或已暂停状态，具体由后续实现决定

#### 交互 4：浏览日期

- 横向滑动日期条
- Home 内容切换到对应日期

#### 交互 5：Calendar 跳转

- 点击 Calendar 按钮
- 弹出月历
- 选择某天
- 切换到对应 Home

#### 交互 6：补录

补录方式：

- 选择日期
- 拖动开始时间
- 拖动结束时间
- 选择任务 / 类别
- 保存

拖动时：

- 带轻震动
- 有时间刻度吸附感

### 8.4 Home 页面状态

#### 空状态

当天没有任务时：

- 显示简洁空状态插图或占位文案
- 按钮引导 Add your first task

#### 有任务但没开始

显示任务列表，状态为未开始。

#### 有任务进行中

显示当前进行中卡片，任务列表也同步状态。

#### 查看历史日期

- 顶部日期不是 Today
- 右下角按钮变 Today
- 内容展示该日记录
- 默认不鼓励在历史日期继续新增任务

## 9. Calendar 弹层 PRD

### 9.1 定位

Calendar 不是一个页面，而是 Home 的日期选择弹层。

### 9.2 打开方式

Home 顶部导航栏右侧 Calendar 按钮点击后弹出。

### 9.3 展现形式

建议：

- bottom sheet
- 或 centered sheet

MVP 更推荐 bottom sheet。

### 9.4 内容结构

顶部：

- 当前月
- 月份切换

主体：

- 月历网格
- 选中日期高亮
- 今天标记
- 有记录的日期可增加轻量标记

行为：

- 选中某天后关闭弹层
- Home 切换到选中日期

## 10. Insights 页面 PRD

### 10.1 页面目标

Insights 用于分析时间分配、项目占比、分类占比和长期趋势。

### 10.2 时间维度

支持：

- Day
- Week
- Month
- Year

#### Day

- 当天项目占比
- 当天分类占比
- 已记录 / 未记录时长

#### Week

- 每天投入变化
- 本周项目分布

#### Month

- 每周投入变化
- 月内项目分布

#### Year

- 每月投入趋势

### 10.3 核心交互

#### 交互 1：切换维度

点击 Day / Week / Month / Year，页面整体更新。

#### 交互 2：点项目看详情

可进入项目详情页，查看：

- 项目总投入时长
- 近期趋势
- 关联任务

这个可以是第二版功能，第一版先不强做详情页。

## 11. 补录页 PRD

### 11.1 页面目标

解决：

- 忘记开始计时
- 忘记结束计时
- 想补上娱乐 / 健身 / 吃饭 / 通勤等时间段

### 11.2 页面结构

顶部：

- 日期选择
- 取消 / 保存

中间：

- 开始时间
- 结束时间
- 可拖动时间轴

下方：

- 选择任务
- 或快速归类：Break / Entertainment / Meal / Commute / Other

最下：

- 保存按钮

### 11.3 交互重点

- 拖动时有 haptic
- 时间吸附到 5 分钟 / 10 分钟节点
- 滑动过程顺畅
- 尽量避免复杂表单感

## 12. 任务与项目逻辑

### 12.1 Project

大的容器，比如：

- FIRE App
- Pet App
- Time App

### 12.2 Task

更细的动作，比如：

- Design Home UI
- Fix tab bar
- Right floating action button

Task 属于某个 Project，也可以允许没有 Project，具体在 MVP 实现时再确定。

### 12.3 Session 与计时边界（规格硬化）

以下规则应在实现前写进工程规格，避免「能跑但数据不可信」：

**实体关系（概念层）**

- 一个 Task 可有多个 Session（每次开始—结束或一次补录对应一个 Session）。
- Session 记录：`taskId`、`projectId`（可空）、`start`、`end`（进行中则 `end` 为空）、`source`（正计时 / 补录）、`pausedDuration`（若采用「暂停累计」模型则需）。

**状态机（正计时）**

建议状态：`idle` → `running` ⇄ `paused` → `ended`（完成或放弃）。从 `running` 进入后台时仍计为 running，除非用户暂停或结束。

```
idle --开始--> running --暂停--> paused --继续--> running
running --结束--> ended
paused --结束--> ended（保存已计时长，规则需明确：是否丢弃暂停段）
```

**必须定义的行为**


| 场景         | 规格要求                                                                       |
| ---------- | -------------------------------------------------------------------------- |
| 跨自然日       | Session 是否允许跨日？若允许，Insights 按「开始日」还是「结束日」归因？需单一默认策略。                       |
| 重叠 Session | 是否允许同一 Task 或不同 Task 在同一时间重叠？默认建议：**不允许重叠**；若检测到重叠，补录时拒绝或合并，正计时中已开始则提示先结束。 |
| App 被系统杀死  | 恢复时：展示「上次计时可能未保存」，提供放弃或按最小时长保存的选项（具体文案可再定）。                                |
| 时区 / 旅行    | 至少定义：Session 存 UTC 还是本地墙钟，历史日视图以何者为准。                                      |


## 13. 视觉方向

### 13.1 液态玻璃使用范围

只用于少量高价值位置：

- tab bar
- 右下角主按钮
- 月历弹层少量顶部区域

不要全页玻璃。

### 13.2 卡片

- 圆角
- 阴影很轻
- 分隔清楚
- 不过度装饰

### 13.3 参考资源

- [awesome-design-md](https://github.com/VoltAgent/awesome-design-md)：DESIGN.md 与设计系统相关资源合集（VoltAgent）

### 13.4 偏好参考产品（体验与视觉气质）

以下为产品负责人主观偏好，用于对齐 App 的整体气质与信息密度，**并非**功能层面的照搬：

- **Apple 健康**：系统级清晰层级；摘要卡 + 圆环/趋势；数据可读优先；图表克制、标签明确。
- **Oura**：柔和配色与 calm 感；日/周叙事；圆环与「分数」式摘要带来的可扫读性（本 App 可对应「今日已记录 / 项目占比」等）。
- **Gentler Streak**：温和、非惩罚性的进度与连续记录；鼓励式语气；避免制造焦虑的「赤字」展示（与本 App「未记录时长提醒」的表达方式可对照）。
- **Notion**：块级信息结构、留白与标题层次；列表与分组的秩序感（可借鉴于任务列表、Insights 区块排版）。

## 14. MVP 开发优先级

（下列顺序已合并「选择性扩展」决策：在保留原主路径前提下，提前 Insights 最小切片、引导、导出与部分体验项。）

### Phase 1

- Home 基础页面
- Task / Project 创建
- 正计时
- session 保存
- 日期滑动条
- 顶部日期 + Calendar 弹层
- **首次使用引导（约 60 秒）**：创建第一个 Project、第一个 Task、完成一次开始/结束计时
- **极简 Insights（仅当日）**：至少展示「当日项目占比」一种视图（可与 Home 当日数据同源，避免重复逻辑）

### Phase 2

- 倒计时
- Insights（在「当日占比」之上补齐周/月/年等规格 §10 中的维度）
- 手动补录
- 未记录提醒
- **重复任务**（由原 Phase 3 提前）
- **深色模式**（跟随系统或 App 内开关，由原 Phase 4 提前）
- **本地备份 / 导出**（如 JSON 或 CSV，便于换机与自查；不上云则仍属隐私友好）

### Phase 3

- 睡眠录入
- haptic 优化
- Today 按钮切换逻辑完善

### Phase 4

- HealthKit
- 小组件

## 15. 最终结论

当前结构收敛为：

- Home：执行 + 记录 + 查看具体某天
- Insights：分析 + 趋势 + 占比
- Calendar：只是日期跳转入口，不单独成页

这是一个合理的产品结构。下一步建议先输出一份 Claude Code / Cursor 可直接用的开发规格说明（含 §12.3 的数据与状态细节），然后再进入 SwiftUI 页面结构草稿。
